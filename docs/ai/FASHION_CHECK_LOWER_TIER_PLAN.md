# 时尚品鉴低贴合档位实施计划

## 当前决定

- 当前网页、`tag-database.json` 和本周公开数据只展示已核验金牌装备，不展示一星或其他低贴合装备。
- `data/fashion-check/lower-tier-candidates.json` 只保存已审阅来源明确列出的候选，不能被公开页面或金牌生成器消费。
- 在分值完成实测前，不使用“银牌 +N”“一星 +N”或“部分匹配 +N”等用户可见结论。
- 低贴合候选不得并入金牌物品数组、金牌数量、80/100 分方案或金牌查询结果。

## 术语约定

不同资料使用的词不统一，采集时保留原词，内部统一分类，展示名称以后由维护者确认。

| 来源用词 | 当前解释 | 当前处理 |
| --- | --- | --- |
| `1 Star` / `one-star` | 社区来源明确列出的低于金牌档装备 | 内部候选档 `one-star`，不推断分值 |
| `partial match` | 对主题存在部分匹配的描述 | 只作为检索别名，不自动等同某个档位 |
| `light-blue stamp` | 对评审印章外观的描述 | 只作为检索/实测记录，不自动换算分值 |
| `Low` | AvantGarde 对游戏 `Score stamp` 低档值的注释 | 只记录原始印章等级，不作为正式中文名 |
| `New Golds` | 周更资料中新增金牌装备的版面分组 | 不是评分档位，不能用于识别低贴合装备 |
| 银牌 | 国内可能使用的俗称 | 在机制和分值未确认前不作为正式数据名 |

采集器必须保存 `rawClaim` 和来源原词。标准化只能增加内部 `tierId`，不能覆盖原始证据，也不能因为命中某个别名就自动接受物品或分值。

## 已知与未知

已确认：

- 穿满参与评审的 11 个装备栏位时，最低基础分为 68；副手、脸饰和灵魂水晶不参与该基础分计算。
- 金牌主题装备通常为左侧装备 `+8`、饰品 `+6`，个别物品可能存在额外 1 分。
- 游戏窗口存在 `Score stamp` 等级；公开参考只明确 `0 = Gold`、`4 = Low`、`5 = None`，没有给出中间等级的稳定名称和分值。
- 已审阅社区帖子会用 `1 Star` 列出非金牌候选，但未提供足以证明统一分值的计分表。

仍需实测：

- `1 Star` 是否与单一 `Score stamp` 值完全对应。
- 一星装备是否固定加分，或根据左侧装备、饰品、具体标签及具体物品变化。
- 同一物品在不同周次复用同一标签时是否保持相同分值。
- 低贴合装备是否也存在额外 1 分的特殊物品。

## 实测方案

### 测试前提

1. 穿满参与评分的 11 个装备栏位，确保没有空位。
2. 目标部位以外的装备保持不变，并避免其他主题命中。
3. 使用未染色装备，排除同色系 `+1` 和精确染色 `+2`。
4. 评审按当前实际外观计算；存在投影时记录投影后的 Item ID，不能只记录装备本体。
5. 每次只替换一个目标部位，禁止同时改变装备、染色或其他评分变量。

### 最小测试步骤

1. 用不匹配目标标签的装备评审一次，记录基准总分和目标部位印章值。
2. 只把目标部位换成候选一星装备，再评审一次。
3. 记录新总分，以 `新总分 - 基准总分` 得到该物品本周实测贡献。
4. 截取或导出评审结果，记录游戏显示的印章外观和可读取的 `Score stamp` 原始值。
5. 优先分别取得至少一组左侧装备和一组饰品样本；发现分值不同时，不得继续使用全局统一分值假设。
6. 条件允许时，在候选再次出现的其他周次复测；特殊加分必须按物品记录，不能写成档位默认值。

每周只有 4 次评审机会。一次基准加一次候选至少消耗 2 次；剩余次数优先用于另一部位类型或异常复测，不为了填满数据牺牲对照条件。

### 实测记录字段

每条有效记录至少包含：

```text
globalIssue / cnIssue
categoryId / slotId
itemId / appearanceItemId
tierLabelRaw / scoreStampRaw
baselineScore / candidateScore / measuredDelta
dyeId
allRequiredSlotsFilled
otherThemeMatches
capturedAt
evidenceFile / reviewerNote
```

`measuredDelta` 必须来自同一角色、同一周次、同一套对照条件。只有总分截图而没有基准条件的记录可以保留为线索，但不能进入已核验分值。

## 数据设计

### 内部候选

继续使用 `data/fashion-check/lower-tier-candidates.json` 保存来源明确报告的候选。候选允许 `points` 为空，且不进入任何公开生成物。

实测完成后新增独立的已核验数据，不改写金牌答案结构。建议主键为：

```text
categoryId + slotId + itemId
```

每条已核验关系包含：

```text
tierId
points
scoreStampRaw
scope: global | slot | item
evidence[]
verifiedAt
```

如果实测表明分值按部位或物品变化，`points` 必须保存在关系级，不能只在 `tierId` 上配置一个全局常量。

### 公开数据

低贴合数据使用独立、可懒加载的公开文件，例如 `fashion-check/lower-tier-database.json`。不要直接扩写现有金牌 `tag-database.json`：

- 金牌查询在低贴合文件不可用时仍能独立工作。
- 可单独撤回低贴合数据，不影响已验证金牌答案。
- checker 可以对低贴合证据、分值和槽位执行更严格的发布门禁。

公开文件只包含页面需要的分类 ID、槽位、Item ID、已核验分值和多语言显示字段；不包含原始帖子正文、本地证据路径、内部备注或未解决候选。

## 页面计划

1. 仅在“金牌查询”中有已核验低贴合数据时加载独立数据文件。
2. 每个标签/部位继续先显示“金牌装备”，低贴合装备作为下方独立板块，不混入金牌物品网格和计数。
3. 板块标题在实测后再确定；若最终采用“一星装备”，同时展示已确认的实际加分，不使用“银牌”推断。
4. 无低贴合数据时不显示空板块、占位提示或解释性文案。
5. 当前周作业页只有在当周来源和分值都已核验时才可展示低贴合替代方案；低贴合数据本身不能自动生成 80/100 分作业。
6. 中英日韩名称继续由官方 Item/FashionCheck CSV 生成；固定 UI 文案通过本地化 key 提供。

## 实施阶段

### 阶段 0：暂停公开

- 保留候选和来源原词。
- 不修改公开 JSON、Vue 页面和金牌统计。
- 等待维护者提交可复核的游戏内分值测试。

### 阶段 1：实测与定档

- 按对照实验记录总分差、印章值和物品外观层 Item ID。
- 至少覆盖左侧装备和饰品；对任何不一致结果单独复测。
- 根据证据决定正式中文名称和分值模型。

### 阶段 2：生成器与 checker

- 建立已核验低贴合关系数据和独立公开生成物。
- 拒绝缺失分值、未知 Item、错槽位、无证据关系、与金牌重复的关系及敏感字段。
- 生成中英日韩名称和图标，不使用名称模糊匹配接受 Item。

### 阶段 3：金牌查询页面接入

- 懒加载低贴合文件并渲染独立板块。
- 保持金牌排序、计数和现有查询行为不变。
- 完成桌面、移动端、四语切换和数据不可用回退验证。

### 阶段 4：发布与回滚

- 先发布独立数据文件，再发布前端；旧前端不会读取新文件。
- 回滚时移除前端低贴合板块或停止发布独立文件，金牌查询仍保持可用。
- 未经维护者确认，不把内部候选批量提升为公开已核验关系。

## 上线门禁

以下条件全部满足后才能开始公开实现：

- 维护者确认正式展示名称。
- 分值已经通过可复核的对照实验确认。
- 已确定分值是全局固定、按槽位不同还是按物品不同。
- 每个公开关系均有官方 Item ID、正确槽位和至少一条有效实测证据。
- 低贴合关系与金牌关系分离，且不会改变金牌计数。
- 生成器、checker、类型检查、本地化检查和构建通过。
- 真实浏览器确认金牌与低贴合板块分隔、四语名称正确、移动端无溢出。
